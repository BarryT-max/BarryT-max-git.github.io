ID Matching Workflow

进士录与 CBDB 人物匹配工作流程

目标：以进士录为基准，将 CBDB 人物尽可能准确地贴合到进士 id，并保留可审计的候选、得分、歧义和冲突记录。

主表口径：进士录 姓名精确匹配 + 年份/朝代验证 输出含候选、选中、歧义与报告表
方法摘要

先清洗两套人物姓名并生成统一姓名键；唯一名对唯一名直接匹配；所有涉及重名的候选再用进士年份、CBDB entry 年份、朝代兼容关系分轮筛选。无法唯一确认的记录保留为 ambiguous，跨记录共享同一 CBDB 人物的记录标为 conflict。

91,749 进士录主表记录数
87.50% 含 conflict 的自动贴合率
84.90% 排除 conflict 的强匹配率

使用的数据

  • JINSHI.dta
  • BIOG_MAIN_Original.dta
  • ENTRY_DATA_Original.dta
  • ENTRY_CODES.dta

姓名怎么统一

清洗姓名，CBDB 繁体转简体，去除空白，生成统一的 match_key。

一条记录代表谁

一个 jinshi_id 代表一个进士记录；一个 c_personid 代表一个 CBDB 人物。

一、整体思路

整体流程

这一部分说明从原始数据到最终进士主表的总体路径。每一步都产生中间文件，便于回溯和人工复核。

1. 清洗两边人物表 生成进士清洗表与 CBDB 人物清洗表，标记姓名唯一、姓名重名、id 重复、缺失姓名。
→
2. 清洗 ENTRY 证据 只抽取进士相关 entry 年份：29/36/37/44。这些代码平等参与匹配。
→
3. 按姓名构造候选 保留 name matched、JINSHI only、CBDB only，并按两边姓名唯一性分层。
4. 唯一名直接匹配 进士姓名唯一且 CBDB 姓名唯一时，直接选中，标记为 name_only_unique_unique。
→
5. 涉及重名则二次验证 按朝代兼容与年份差分轮筛选：精确年份、±10 年、±50 年。
→
6. 歧义与冲突单独保留 任何涉及重名的类别，只要无法唯一选出 CBDB 人物，都进入 ambiguous；同一 CBDB 人物匹配多个进士标记 conflict。

匹配策略与步骤

下面按实际数据操作顺序展开，重点说明在哪一步清洗、在哪一步按姓名合并、在哪一步加入年份证据，以及最终如何把 CBDB 数据贴回进士录。

步骤 具体操作 生成/使用的表
0. 清洗输入 清洗 JINSHI.dta 的姓名、朝代、年份，生成 jinshi_id、match_key、jinshi_year、jinshi_dy，并标记姓名是否唯一。 IDMatch_v3_jinshi_clean.dta
0. 清洗输入 清洗 BIOG_MAIN_Original.dta，将 c_name_chn 转简体，生成 match_key，保留 c_personid、c_dy、c_index_year、生卒年，并标记姓名是否唯一。 IDMatch_v3_cbdb_person_clean.dta
0. 清洗 ENTRY 从 ENTRY_DATA_Original.dta 中保留 c_entry_code in 29,36,37,44 的记录和有效 c_year。这些是进士相关 entry，地位平等。 IDMatch_v3_entry_jinshi_evidence.dta
1. 第一轮唯一名匹配 先取进士中 jinshi_name_unique == 1 的记录；再取 CBDB 中 cbdb_name_unique == 1 的记录；以进士表为基准，按 match_key 做 merge 1:1。_merge == 3 的记录直接选中。 进入 IDMatch_v3_selected_matches.dta，标记 match_round = 0 和 final_match_type = name_only_unique_unique。
2. 保留第一轮未匹配者 _merge == 1 是唯一名进士但 CBDB 没有唯一名匹配；_merge == 2 是唯一名 CBDB 但进士侧没有唯一名匹配。它们不丢弃，会进入后续候选分类或覆盖率报告。 记录在 IDMatch_v3_name_candidates_all.dta。
3. 构造涉及重名的候选 分别处理三类：进士重名 × CBDB 唯一名、进士唯一名 × CBDB 重名、进士重名 × CBDB 重名。三类都必须经过年份和朝代二次验证；即使前两类只有一边重名，也可能因为年份缺失或并列最佳而无法自动选出唯一匹配。 先进入 IDMatch_v3_name_candidates_all.dta，再压缩成候选对。
4. 加入年份证据 把候选对按 c_personid 连接到 IDMatch_v3_entry_jinshi_evidence.dta，先计算 abs(jinshi_year - c_year)。如果可用 c_year 不足以判断，再使用 abs(jinshi_year - c_index_year) 作为补充时代证据。 IDMatch_v3_candidate_pairs_scored.dta
5. 重名二次筛选 只对未在第一轮选中的进士操作。按朝代兼容 + 年份规则依次筛：精确年份、±10 年、±50 年。三类重名候选都执行同一规则；若同一 jinshi_id 只有一个最佳 c_personid，选中；若没有年份证据或最佳候选并列，进入 ambiguous。 选中者进入 IDMatch_v3_selected_matches.dta；并列者进入 IDMatch_v3_ambiguous_candidates.dta。
6. 输出进士主表 以进士录清洗表为基准，按 jinshi_id 连接最终选中匹配和 CBDB 人物字段，把 c_personid、CBDB 姓名、朝代、年份等贴回每条进士记录。 IDMatch_v3_jinshi_with_cbdb.dta
二、匹配规则

年份+朝代模糊匹配规则

进士录中的“公元年”和 CBDB 进士相关 entry 的 c_year 语义最接近，因此优先用 c_year 验证重名候选；c_index_year 只作为 c_year 不可用时的补充时代证据。

证据 ENTRY 条件 用途
进士相关入仕年份 c_entry_code in 29, 36, 37, 44 这些代码按人工整理结果平等处理，均可作为进士相关 c_year 证据。
入仕年份差 diff_entry_year = abs(jinshi_year - c_year) 这是重名筛选的主年份证据，因为进士录“公元年”和 CBDB entry 入仕年份语义相同。当前数据中进士相关 entry 共 98,366 条，其中 88,363 条有有效 c_year，有效率 89.83%。
人物索引年份差 diff_index_year = abs(jinshi_year - c_index_year) 作为补充时代证据。按人物统计，96,239 个有进士相关 entry 的 CBDB 人物中，87,971 人有有效 c_year，有效率 91.41%；缺 c_year 但有 c_index_year 的人物有 1,585 人，占 1.65%。

重名处理规则

所有涉及重名的候选都必须经过二次验证。无论是“进士重名 × CBDB 唯一名”、还是“进士唯一名 × CBDB 重名”、或“两边都重名”，只要不能唯一确认，就不强行归入强匹配。

轮次 规则 选择原则
Round 1 朝代兼容 + 年份完全相同 若同一 jinshi_id 只有一个最佳 c_personid，选中；否则 ambiguous。
Round 2 朝代兼容 + 年份差 ≤ 10 只处理前一轮未匹配者，仍按最佳年份差选一人。
Round 3 朝代兼容 + 年份差 ≤ 50 作为宽松补充，选中结果在 IDMatch_v3_selected_matches.dta 和 IDMatch_v3_candidate_pairs_scored.dta 中用 match_round、match_rule、year_source 单独标记；无法唯一选择的候选进入 ambiguous。

匹配轮次解释

match_round 记录一条自动匹配是在哪一轮被选中的。数值越靠前，证据通常越强；match_round = . 表示没有进入自动选中结果。

变量/状态 含义 解释
match_round = 0 唯一名直接匹配 进士唯一名 × CBDB 唯一名，按姓名直接选中，不要求年份二次验证。
match_round = 1 c_year 精确匹配 重名候选中，进士录 公元年 与 CBDB 进士相关 entry 的 c_year 完全相同。
match_round = 2 c_year 差值 ≤ 10 上一轮未匹配者中，使用 c_year 小范围年份匹配。
match_round = 3 c_year 差值 ≤ 50 上一轮未匹配者中，使用 c_year 中范围年份匹配。
match_round = 4 c_index_year 精确兜底 仅当 c_year 不可用时，使用 c_index_year 作为补充时代证据。
match_round = 5 c_index_year 差值 ≤ 10 c_year 不可用时的 index year 小范围兜底。
match_round = 6 c_index_year 差值 ≤ 50 c_year 不可用时的 index year 中范围兜底。
match_round = . 未自动选中 没有进入最终 selected，通常出现在 unmatched 或 ambiguous 候选中。

匹配状态说明

这四个状态用于区分“已自动匹配”“自动匹配但存在跨记录冲突”“有候选但无法唯一确认”和“没有自动匹配结果”。

状态 进入主表的方式 是否建议直接当强匹配
selected 候选对先在 IDMatch_v3_candidate_pairs_scored.dta 中通过某轮规则，被标记为 final_selected = 1，再进入 IDMatch_v3_selected_matches.dta，最后按 jinshi_uid 贴回进士主表。 可以作为普通自动匹配使用；若同时没有 conflict，则 strong_match = 1。
conflict 先被自动选中并进入 selected_matches；随后程序发现同一个 c_personid 被多个 jinshi_uid 选中，于是把状态改为 conflict，再贴回主表。 不建议直接当强匹配；它满足单条进士记录的筛选机制，但存在跨记录重复占用，需要复核。主表中已经设置 strong_match = 0。
ambiguous 在单条进士记录内部无法唯一选出一个 CBDB 人，或所有规则后仍不能确认。它进入 IDMatch_v3_ambiguous_candidates.dta，不作为强匹配贴入主表。 需要人工判断。
unmatched 最终没有自动匹配结果。可能完全没有姓名候选，也可能有 ambiguous 候选但未被自动选中。 不能当匹配使用。

当前主表状态分布：selected 77,891 条，conflict 2,392 条，unmatched 11,466 条。若把 conflict 也算“已贴上 CBDB 但需复核”，匹配率为 87.50%；若只算非 conflict 的强匹配，匹配率为 84.90%。主表中已经加入 strong_match，定义为 final_match_status == "selected" 且不是 conflict。

三、输出结果

输出文件说明

输出文件分为三类：清洗输入表、匹配过程表、最终结果表。最终主表是进士录口径，即把 CBDB 信息贴回每条进士记录。

文件 作用
IDMatch_v3_jinshi_clean.dta进士录清洗表。
IDMatch_v3_cbdb_person_clean.dtaCBDB 人物清洗表。
IDMatch_v3_entry_jinshi_evidence.dta进士相关 ENTRY 年份证据表。
IDMatch_v3_name_candidates_all.dta按姓名生成的全部候选与 only 记录。
IDMatch_v3_candidate_pairs_scored.dta一行一个进士-CBDB 候选对，含年份、朝代、轮次和得分。
IDMatch_v3_selected_matches.dta最终自动选中的匹配对。
IDMatch_v3_ambiguous_candidates.dta无法自动唯一判断的候选。
IDMatch_v3_jinshi_with_cbdb.dta进士录层最终主结果，把 CBDB 数据贴到进士记录上。
IDMatch_v3_match_report.dta随数据保存的统计报告表，每一轮记录输入数、候选数、选中数、ambiguous 数、损失数。

report 表说明

IDMatch_v3_match_report.dta 是随数据保存的流程统计表，不是人物数据本身。它记录每一步的输入规模、候选数量、选中数量、歧义数量和剩余数量，用来回答“每一轮损失了多少”。

字段 含义
step当前流程步骤。
rule当前步骤使用的规则。
input_n这一阶段的输入规模。
candidate_n这一阶段符合条件的候选数量。
selected_n这一阶段自动选中的数量。
ambiguous_n这一阶段进入歧义或并列的数量。
remaining_n这一阶段后仍未匹配或未解决的数量。

ambiguous 和未匹配

ambiguous 不等于完全没有候选。它表示有同名候选，但自动规则无法唯一确认；未匹配则是最终主表中没有贴上 CBDB 人物。

口径 数量 含义
最终未自动匹配 11,466 最终进士主表中没有贴上 c_personid 的记录。
未匹配但有 ambiguous 候选 5,368 有同名候选，但由于年份缺失、朝代/年份规则不足、或候选并列，程序没有强行选择。
未匹配且无 ambiguous 候选 6,098 基本是姓名层面没有可用 CBDB 候选的进士记录。
存在 ambiguous 候选的进士 5,505 候选表口径；其中少量进士同时也有另一个候选被选中，所以这个数略大于“未匹配但有 ambiguous 候选”。

后续清理版将在 ambiguous 表中增加 jinshi_already_selected：若同一个 jinshi_uid 已经在 selected 表中有最终匹配，则该变量为 1；否则为 0。这样可以区分“最终未匹配、需要人工处理的 ambiguous”和“已有自动匹配但仍保留其他可疑候选的 ambiguous”。
